## High-value fixes (before ship)

### 1. HSL fallback missing from CSS export — violates FR-16 (must-have)
`src/components/export/ExportModal.tsx:129-269` — both `generateCssVariables` and `generateTailwindConfig` only emit OKLCH. PRD explicitly requires:
```css
@supports not (color: oklch(0% 0 0)) {
  :root { --color-bg: hsl(...); ... }
}
```
The HSL values are already computed and available on every `ColorLevel`/`AccentShade` as `css.hsl`. Wiring is trivial.

### 2. HueWheel is mouse-only → broken on tablet/mobile
`src/components/wizard/HueWheel.tsx:72-101` uses `onMouseDown`, `window.mousemove`, `window.mouseup`. No touch or pointer events. PRD non-functional requirement: "Usable on tablet and desktop." Replace with `onPointerDown` / `pointermove` / `pointerup` + `setPointerCapture`.

### 3. Zero responsiveness — hardcoded 60/40 split
`src/app/page.tsx:97,99,202` uses `flex` + `w-[60%]` + `w-[40%]` with no breakpoints. Below ~768px the right panel collapses to a 40% sliver and the wizard controls overflow. Needs:
- `flex-col md:flex-row`
- Full-width children on small screens
- Preview either stacks below or moves to a tab
- Header and step nav should stay usable at narrow widths

### 4. Contrast indicator isn't a persistent bottom bar
PRD FR-13: "Persistent WCAG contrast indicator (bottom bar)."
Current implementation (`LivePreview.tsx:60-71`) is a panel *inside* the right preview column — scrolls with the preview, disappears at narrow widths, and only checks a **single** combination (`text.primary vs levels.l0`).

Should also validate:
- `text.secondary vs l0`
- `text.muted vs l0` (expected to fail AA — that's its purpose, but should be shown as "AA Large" status)
- `text.primary vs l1` (cards)
- Primary accent button (base shade vs its derived text shade)

### 5. Dead wizard components
`src/components/wizard/PrimaryAccentStep.tsx` and `src/components/wizard/SecondaryAccentStep.tsx` are both unreferenced. `page.tsx` uses the combined `AccentStep.tsx`. Delete both.

### 6. Lock file conflict
`package-lock.json` is committed, `pnpm-lock.yaml` is untracked (per `git status`). Pick one:
- If deploying to Vercel, npm is the simplest path (already committed)
- If pnpm is preferred, delete `package-lock.json` and commit `pnpm-lock.yaml`

Mixed lock files cause CI-versus-local drift.

## Medium-value (quality of life)

### 7. Stack version drift vs PRD
PRD calls for Next.js 15 + React 19. Actual:
- `next@14.2.35`, `react@18.3.1`, `react-dom@18.3.1`
- But `eslint-config-next@16.1.6` (meant for Next 16)
- And `tsconfig.json` includes `".next/dev/types/**/*.ts"` (Next 15+ path)

Everything still builds. To get internally consistent, either:
- Upgrade to Next 15 + React 19 (matches PRD, enables React Compiler, modern features)
- OR pin `eslint-config-next` to `^14.2.0` and drop the Next 15 tsconfig path

### 8. Stale `lightPalette` derivation in `page.tsx:41-54`
```ts
const lightPalette = useMemo(() => generateModePalette('light', lightL0, scale), ...);
const darkPalette = useMemo(() => generateModePalette('dark', darkL0, scale, lightPalette.levels), ...);
const lightPaletteWithText = useMemo(() => generateModePalette('light', lightL0, scale, darkPalette.levels), ...);
```
`lightPalette` exists only to break the circular text-color dependency, then is discarded. Its `text` values are technically stale (computed against a default dark palette) but nothing reads them. Cleaner: compute `levels` once, then compute `text` for both modes after both `levels` exist.

### 10. `findContrastingLightness` is dead code
`src/lib/color/contrast.ts:116-148` — computes `bgLuminance` then never uses it (binary searches directly against `getContrastRatio`). Not imported by any UI. Either delete or actually wire it into the contrast indicator to suggest a fix when a combo fails.

### 11. Range-locked HueWheel gradient math
`src/components/wizard/HueWheel.tsx:114-123`:
```ts
background: `conic-gradient(from ${minHue - 90}deg,
  oklch(65% 0.2 ${minHue}),
  oklch(65% 0.2 ${(minHue + maxHue) / 2}),
  oklch(65% 0.2 ${maxHue}),
  oklch(30% 0.05 0) ${maxHue - minHue}deg
)`
```
The stop positions don't actually describe an arc of `maxHue - minHue` degrees — the first three stops are distributed across the full 360° circle (no explicit positions), then a hard stop is placed at `maxHue - minHue deg` after. The clamping logic in `calculateHue` still works, so it's functional, but the visual doesn't match the allowed range. Worth fixing if users find it confusing.

### 12. Single-combination contrast checking
Already mentioned in #4 but worth calling out separately. Currently:
- `page.tsx:82-90` checks `text.primary vs levels.l0` only
- Should cover the full matrix, at least: primary/secondary/muted text × l0/l1 background, plus accent button contrast

### 13. Initial L0 defaults
`page.tsx:29-30` hardcodes `lightL0=100` and `darkL0=0` (pure white, pure black). PRD examples use L97/L3. Pure endpoints are valid but boring; consider defaults like L97 / L3 tied to the initial scale.

export const viewport: Viewport = { themeColor: '#000', width: 'device-width' };

## Notes / context for whoever picks this up

- The color math library (`src/lib/color/`) is solid. OKLCH → HSL via culori, WCAG via `wcagContrast`. Not a source of bugs.
- The wizard state lives entirely in `page.tsx` as individual `useState` calls. For current scope this is fine, but if the user wants URL sharing (FR-17, nice-to-have) it'll need to be serialized to search params — consider a single config object up front.
- `SYSTEM_ACCENT_DEFAULTS` in `src/lib/color/accents.ts` is the source of truth for the hue ranges used to constrain the system accent wheels.
- No API routes, no server actions, no env vars — pure static SPA. Vercel deploy will be fast.
